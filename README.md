# 客服反馈智能体 · 项目说明

## 一句话说清楚

> 用"**规则优先、AI 兜底**"的方式，让计算机自动处理客服工单。80% 的常见问题靠规则秒回，20% 的疑难问题才交给大模型思考。

---

## 为什么要做这个

读过王秋禾《增能视角下"社工+志愿者"协同模式》论文后，发现论文提出的方法论，恰好能解决当前智能客服的三大毛病：

| 当前客服系统的毛病 | 论文中的对应问题 |
|-------------------|-----------------|
| 什么都丢给 AI，回答飘忽不定 | 响应机制滞后 |
| 简单问题也要等很久 | 资源供给短缺 |
| 复杂问题各部门互相推诿 | 协同联动不足 |

**核心思路**：把论文中"社工 + 志愿者"的分工逻辑，搬到计算机系统里。

---

## 它是怎么工作的（文科版）

想象一个**社区服务中心**：

```
                    ┌─────────────┐
                    │   社工老王    │  ← 调度中心，决定谁来处理
                    │  (总调度)    │
                    └──────┬──────┘
                           │
       ┌───────────────────┼───────────────────┐
       │                   │                   │
  ┌────▼────┐      ┌──────▼──────┐     ┌──────▼──────┐
  │ 小王    │      │   资料室    │      │   专家顾问   │
  │ 前台接待 │      │  档案+政策  │      │  (AI大模型)  │
  │         │      │            │      │             │
  │ 一听就   │      │ 查FAQ      │      │ 只在以下情况 │
  │ 知道咋办 │      │ 查政策条款  │      │ 才会请出来： │
  │ 的问题   │      │ 查历史案例  │      │ • 大家都不   │
  │         │      │            │      │   知道答案   │
  │ • 你要   │      │            │      │ • 几个人意见 │
  │   退款   │      │            │      │   不一致     │
  │ • 你很   │      │            │      │ • 你情绪很   │
  │   生气   │      │            │      │   激动       │
  │ • 这很   │      │            │      │             │
  │   紧急   │      │            │      │             │
  └────┬────┘      └──────┬──────┘     └──────┬──────┘
       │                   │                   │
       └───────────────────┼───────────────────┘
                           │
                    ┌──────▼──────┐
                    │   生成回复   │
                    │  (拼装答案)  │
                    └─────────────┘
```

### 分三步走

**第一步：8 个"前台接待"同时判断**

用户发来一条消息，8 个专门的小模块同时工作：

| 小模块 | 它做什么 | 打个比方 |
|--------|---------|---------|
| 意图识别 | 你想干嘛？退款/投诉/咨询/报修？ | 前台听你第一句话就知道该给你拿什么表 |
| 实体提取 | 订单号多少？多少钱？哪天买的？ | 前台从你话里抓关键信息填表 |
| 情感判断 | 你生气了吗？有多生气？ | 前台看你脸色，决定要不要给你倒杯水 |
| 紧急程度 | 这事有多急？是投诉到工商那种吗？ | 前台判断要不要叫经理出来 |
| 政策匹配 | 这种情况公司规定是什么？ | 翻规章手册找对应条款 |
| FAQ 检索 | 这个问题之前被问过一百遍了吗？ | 翻常见问题小册子找标准答案 |
| 相似案例 | 以前有没有人遇到过一样的事？ | 翻历史档案看上次怎么处理的 |
| 状态追踪 | 这是你第几次说这事了？还缺什么信息？ | 记事本翻到你这页，看之前聊到哪了 |

**第二步：调度中心做决定**

社工老王（调度中心）看完 8 个人的纸条，分三种情况：

- **情况 A（约 60%）**：8 个人意见一致，都很有把握 → **直接回复，不打扰 AI**
- **情况 B（约 20%）**：大部分人知道答案，个别拿不准 → 用模板拼装回复，必要时请 AI 帮忙润色
- **情况 C（约 20%）**：大家都不确定 / 互相矛盾 / 用户非常激动 → **请 AI 大模型出马**

**第三步：生成最终回复**

不管走哪条路，最终都拼成一条完整的回复，发回给用户。

## 它和论文有什么关系

论文提出了"社工 + 志愿者"协同治理的四大路径，本项目每条都对应上了：

| 论文路径 | 项目实现 |
|---------|---------|
| **优化资源配置** | "前台接待"干 80% 的活，AI 专家只处理 20% 疑难杂症，不浪费 |
| **机制创新** | 从收到消息 → 判断 → 分配 → 回复，全程自动化，不用人盯 |
| **精准补位** | 调度中心负责"该谁管"，8 个模块各管一摊，不会互相踢皮球 |
| **专业+协同** | 每个模块只做自己擅长的一件事，合起来就变聪明了 |

论文中解决的是"社区治理"问题，本项目把同一套方法论用来解决"客服系统"问题——本质是一样的：**把专业判断和落地执行分开，让对的人做对的事**。

## 技术实现（一句话版）

- 规则判断的部分用 **Python** 写（因为它处理文字最方便）
- 最终对外服务的部分用 **Java** 写（因为它稳定、适合企业用）
- 两者之间通过接口通信，就像两个部门之间发邮件

---

## 和市面上"AI 客服"有什么不同

| | 普通 AI 客服 | 本项目 |
|------|------------|--------|
| 处理方式 | 每条消息都丢给 AI | 先走规则，搞不定再叫 AI |
| 回复速度 | 依赖 AI 响应，可能慢 | 规则判断毫秒级，大部分秒回 |
| 成本 | 每条消息都烧钱 | 80% 的消息零成本 |
| 可解释性 | AI 的回答说不清为什么 | 每条回复都能追溯"是哪个规则决定的" |
| 确定性 | 同样问题可能回答不一样 | 规则判断每次都一样 |
